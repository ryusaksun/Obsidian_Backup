## Markdown 格式化规则

## 格式化核心目的

- 格式化的核心目的是让文件内容在 Markdown 语法格式下更易读，所以重点在于格式的修改，除非特别要求，否则内容不能修改。

### 标题规范

- 正文不设 H1 标题: 由于 Obsidian 默认显示文件名（Inline Title），正文中禁止出现 `#` 级别的标题，以避免视觉重复
- 严格层级: 使用 `##` 作为章/大节标题，`###` 或列表作为子节

### 内容提炼

- 去冗余: 彻底清除笔记中无关的数字乱码、符号（如 11111、+4、77777 等）
- 无加粗: 除非特别要求，否则不使用 `**加粗**`，保持页面纯净
- 去除参考链接和参考文献的部分: 删除文件末尾的 URL 链接列表

### 标点与习惯

- 标点清理: 清理掉不必要的、重复的标点
- 句末标点：该添加的地方要添加

### 结构排版

- 拒绝 YAML: 不需要 Front Matter（YAML 区域，即 `---` 包围的内容）
- 拒绝 TOC: 不需要目录（Table of Contents）

### 内容修改原则

- 除非特别说明，否则不能随意修改原始内容: 重点在于格式的修改，保持原始语义和内容的完整性

### 格式化标记

- 完成格式化后，在笔记的末端添加已格式化标记:
    ```markdown
    ---

    **<font color="#2ecc71">✅ 已格式化</font>**
    ```
- 除非特殊要求，否则含有已格式化标记的笔记不需要再进行格式化

### Obsidian 链接

- 保留 `[[WikiLink]]` 格式的内部引用，不要删除我引用的图片。
- 保留图片嵌入语法前的 `!` 符号，即 `![[图片名.png]]` 不能改为 `[[图片名.png]]`
- 示例: `[[直方图均衡化]]` 链接到其他笔记
- 示例: `![[Pasted image 20260107174717.png]]` 保持原样

### 空行与段落规范

- 标题前后保留空行
- 段落之间使用空行分隔
- 列表项之间紧凑（除非子列表）

### 列表格式规范

- 统一使用 `-` 作为列表符号
- 子列表缩进使用 4 空格或 2 空格
- 避免混用有序和无序列表

### 代码块规范

- 指定语言标识（如 ```python）
- 代码行数超过 20 行使用折叠块
- 移除不必要的空行
- 修复没有正确渲染的 bash 代码块（检查是否缺少 ```bash 标记）

### 中英文混排

- 英文单词/数字与中文之间加空格
- 示例: `图像处理是计算机视觉`

### 表格规范

- 对齐方式统一（左对齐优先）
- 列标题使用加粗或区分样式
- 避免过宽的表格（考虑拆分）

### 公式格式

- 行内公式使用 `$...$`
- 独立公式使用 `$$...$$`
- 公式后不加标点

#### LaTeX 公式的颜色设置

**推荐方式：使用 LaTeX 原生的 `\color{...}` 命令**

- 行内公式: `$\color{#00b0f0}{\epsilon}$` 或 `$\color{blue}{x_t}$`
- 独立公式: `$$\color{#00b0f0}{\mathcal{N}(0, \mathbf{I})}$$`
- 预定义颜色名: `black`, `white`, `red`, `green`, `blue`, `yellow`, `cyan`, `magenta`
- 支持十六进制颜色: `\color{#00b0f0}{...}`

**混合使用（文字和公式都加颜色）**:
```markdown
<font color="#00b0f0">这里的</font> $\color{#00b0f0}{\epsilon}$ <font color="#00b0f0">包含了所有随机性</font>
```

**禁止方式**:
- **font 标签不得包裹 LaTeX 公式**：确保 `<font color="...">` 标签不包含在 `$...$` 或 `$$...$$` 之间，否则 LaTeX 渲染会失效
  - 正确示例：`<font color="#00b0f0">这是重要概念</font>，公式为 $\color{#00b0f0}{\epsilon}$`
  - 错误示例：`<font color="#00b0f0">公式 $\epsilon$ 无法渲染</font>`

## 文件处理工作流程

格式化 Markdown 文件时，按以下步骤执行:

0. **格式化前检查**: 在格式化任何新文件之前，必须先检查项目结构是否发生变化
    - 使用 `glob` 工具查找所有 `**/*.md` 文件
    - 与 AGENTS.md 中记录的项目结构对比
    - 如果发现新的目录、新的文件或文件位置变化，必须先更新 AGENTS.md 中的项目结构
    - 更新已格式化文件列表（添加新文件，调整文件路径）

0.5 **检查 LaTeX 渲染问题**: 格式化新笔记前，先通过 git diff 检查是否有 font 标签导致 LaTeX 无法渲染的问题
    - 使用 `git diff --name-only` 查看是否有新的 Markdown 文件变更
    - 搜索新增的笔记文件中的 `<font[^>]*>\s*\$\|<font[^>]*>.*\$.*\$.*\$.*\$.*\$` 或 `</font[^>]*>\s*\\mathcal|<font[^>]*>.*\\mathbf`
    - 如果发现 font 标签包裹了 LaTeX 公式，必须修复
    - 修复方法：将 font 标签拆分到 LaTeX 公式外部，例如：
      - 错误：`<font color="#00b0f0">这里的 $\epsilon$ 包含了从第 1 步到第 $t$ 步叠加的所有随机性</font>`
      - 正确：`<font color="#00b0f0">这里的</font> $\epsilon$ <font color="#00b0f0">包含了从第 1 步到第</font> $t$ <font color="#00b0f0">步叠加的所有随机性</font>`
    - 只有确认所有 LaTeX 公式都能正确渲染后，才开始格式化步骤

0. **Git 提交备份**: 在对新增笔记进行格式化前，先把仓库提交到远程

    - 执行命令: `git push`

    - 使用 `git diff --name-only` 查看是否有新的 Markdown 文件变更
